# M1.4.5: WebSocket Server Implementation - API Documentation

## Overview

The WebSocket server provides real-time bidirectional communication for:
- **Streaming task execution** with progress updates
- **Live query results** with event-based updates  
- **Health monitoring** over persistent connections

## Architecture

WebSocket runs on the same port as the HTTP server (default: 3000). Clients connect via `ws://localhost:3000/ws/*` and receive JSON-encoded events.

## Endpoints

### `/ws/execute` - Stream Task Execution

**Connection:** `ws://localhost:3000/ws/execute`

**Message Format (Client → Server):**
```json
{
  "id": "msg-123",
  "msg_type": "subscribe",
  "channel": "execute",
  "payload": {
    "task": "create a new PDF parser",
    "priority": "normal",
    "timeout": 300,
    "format": "json"
  }
}
```

**Event Format (Server → Client):**
```json
{
  "id": "msg-123",
  "event_type": "start",
  "channel": "execute",
  "timestamp": "2026-04-18T10:30:00Z",
  "data": {
    "message": "Starting task execution",
    "status": "started"
  }
}
```

**Event Types:**
- `start` - Task execution started
- `progress` - Execution progress update (includes percent and message)
- `result` - Execution result available
- `error` - Execution error occurred
- `complete` - Execution finished

**Example Progress Event:**
```json
{
  "id": "msg-123",
  "event_type": "progress",
  "channel": "execute",
  "timestamp": "2026-04-18T10:30:05Z",
  "data": {
    "percent": 50,
    "message": "Processing step 2/4"
  }
}
```

### `/ws/query` - Stream Query Results

**Connection:** `ws://localhost:3000/ws/query`

**Message Format (Client → Server):**
```json
{
  "id": "msg-456",
  "msg_type": "subscribe",
  "channel": "query",
  "payload": {
    "query": "what are the available skills?",
    "limit": 10,
    "format": "json"
  }
}
```

**Event Types:**
- `start` - Query processing started
- `progress` - Query processing progress
- `result` - Result batch received (streams multiple result events)
- `error` - Query error
- `complete` - All results delivered

### `/ws/health` - Health Check over WebSocket

**Connection:** `ws://localhost:3000/ws/health`

**Message Format:**
```json
{
  "id": "health-1",
  "msg_type": "ping",
  "channel": "health",
  "payload": {}
}
```

**Response (pong):**
```json
{
  "id": "health-1",
  "event_type": "pong",
  "channel": "health",
  "timestamp": "2026-04-18T10:30:00Z",
  "data": {
    "status": "connected"
  }
}
```

## Validation Rules

### Execute Request
- `task`: Required, 1-10,000 characters
- `priority`: Optional, one of: low, normal, high (default: normal)
- `timeout`: Optional, 1-3600 seconds (default: 300)
- `format`: Optional, one of: text, json, yaml (default: json)

### Query Request
- `query`: Required, 1-10,000 characters
- `limit`: Optional, 1-1000 (default: 20)
- `format`: Optional, one of: text, json, yaml (default: json)

### Error Responses
```json
{
  "id": "msg-789",
  "event_type": "error",
  "channel": "execute",
  "timestamp": "2026-04-18T10:30:00Z",
  "data": {
    "error_code": "WS_ERROR",
    "error_message": "task field is required and must be non-empty"
  }
}
```

## Message Types

### Client Message Structure
```typescript
{
  id: string;              // Unique message identifier for correlation
  msg_type: string;        // "subscribe", "unsubscribe", "ping"
  channel: string;         // "execute", "query", "health"
  payload: object;         // Request-specific data
}
```

### Server Event Structure
```typescript
{
  id: string;              // Matches client message ID for correlation
  event_type: string;      // "start", "progress", "result", "error", "complete", "pong"
  channel: string;         // Which channel event came from
  timestamp: string;       // ISO 8601 timestamp
  data: object;            // Event-specific data
}
```

## Deployment

### Starting the WebSocket Server

```bash
# Start with default port (3000)
mimi http

# Start with custom port
mimi http --port 8080

# WebSocket available at ws://localhost:8080/ws/execute
```

### Configuration

Add to `~/.config/mimi/config.toml`:

```toml
[server]
http_port = 3000
ws_enabled = true
```
