## Overview

The authentication system provides unified, token-based authentication across CLI, HTTP, and WebSocket interfaces. It uses JWT tokens for stateless authentication and implements role-based access control (RBAC) with three roles: Admin, User, and Guest.

### Key Features

- **JWT Token-based Auth**: Stateless, scalable authentication
- **Role-Based Access Control**: Admin, User, Guest roles with granular policies
- **Cross-Interface Integration**: Works seamlessly with CLI (--token flag), HTTP (Bearer token), and WebSocket (token handshake)
- **Policy Enforcement**: Resource/action-based permission checking
- **Optional Auth**: Endpoints support both authenticated and unauthenticated access

## Architecture

### Token Flow

```
1. Client authenticates (via CLI flag, HTTP header, or WebSocket handshake)
2. AuthManager validates credentials/identity
3. TokenService generates JWT token with claims (sub, username, roles, exp)
4. Token returned to client
5. Client includes token in subsequent requests
6. Middleware/Extractors validate token and extract Identity
7. Handlers check permissions via AuthManager.check_permission()
```

### Role Hierarchy

- **Admin**: Full access to all resources and actions
- **User**: Read/write access to execute and query resources
- **Guest**: Read-only access to query resource

### Types

#### Identity

```rust
pub struct Identity {

```

#### Role
```rust
pub enum Role {
    Admin,

}
```

#### AuthToken (response from token generation)
```rust
pub struct AuthToken {
    pub token: String,           // JWT token string
    pub token_type: String,      // Always "Bearer"
    pub expires_in: i64,         // Seconds until expiration
    pub issued_at: i64,          // Unix timestamp
}
```

#### TokenClaims (inside JWT)
```rust
pub struct TokenClaims {
    pub sub: String,             // User ID
    pub username: String,
    pub roles: Vec<String>,      // Role names
    pub iat: i64,                // Issued at timestamp
    pub exp: i64,                // Expiration timestamp
}
```

### AuthManager API

#### `new(secret: String, token_ttl: i64) -> AuthManager`
Create a new auth manager.

- **Parameters**:
  - `secret`: Secret key for JWT signing (use strong, random value)
  - `token_ttl`: Token time-to-live in seconds

- **Example**:
```rust
let manager = AuthManager::new(
    "your-secret-key-here".to_string(),
    3600 // 1 hour
);
```

#### `generate_token(identity: &Identity) -> Result<String, AuthError>`
Generate JWT token for identity.

- **Returns**: JWT token string or AuthError

- **Example**:
```rust
let identity = Identity {
    user_id: "user123".to_string(),
    username: "john".to_string(),
    roles: {
        let mut set = HashSet::new();
        set.insert(Role::User);
        set
    },
};

let token = manager.generate_token(&identity)?;
```

#### `validate_token(token: &str) -> Result<Identity, AuthError>`
Validate JWT token and extract identity.

- **Parameters**:
  - `token`: JWT token string

- **Returns**: Identity or AuthError (TokenExpired, InvalidToken, RoleNotFound)

- **Example**:
```rust
let identity = manager.validate_token("eyJ0eXAiOiJKV1QiLCJhbGc...")?;
println!("User: {}", identity.username);
```

#### `check_permission(identity: &Identity, resource: &str, action: &str) -> Result<bool, AuthError>`
Check if identity has permission for resource/action.

- **Parameters**:
  - `identity`: User identity
  - `resource`: Resource name (e.g., "execute", "query", "config")
  - `action`: Action name (e.g., "read", "write", "delete")

- **Returns**: Boolean or AuthError

- **Example**:
```rust
if manager.check_permission(&identity, "execute", "write")? {
    // Proceed with write operation
}
```

#### `register_default_policies()`
Register built-in RBAC policies.

- **Registers**:
  - `admin_execute`: Admin can execute any action
  - `user_execute`: User can execute read operations
  - `admin_query`: Admin can query all data
  - `user_query`: User can query all data
  - `guest_query`: Guest can query (read-only)

## Interface Integration

### CLI Usage

Generate token:
```bash
mimi exec "generate token for admin" --token-user admin --token-roles admin
```

Use token in commands:
```bash
mimi exec "list all tasks" --token "eyJ0eXAiOiJKV1QiLCJhbGc..."
```

Or via environment variable:
```bash
export MIMI_TOKEN="eyJ0eXAiOiJKV1QiLCJhbGc..."
mimi exec "list all tasks"
```

### HTTP API

All endpoints support Bearer token authentication:

```bash
curl -X POST http://localhost:8080/api/execute \
  -H "Authorization: Bearer eyJ0eXAiOiJKV1QiLCJhbGc..." \
  -H "Content-Type: application/json" \
  -d '{"task": "list tasks"}'
```

Endpoints can be unauthenticated (optional auth):
```bash
curl http://localhost:8080/health
```

### WebSocket API

Token passed in initial connection or first message:

```javascript
// Option 1: URL parameter (not recommended for security)
ws = new WebSocket("ws://localhost:8080/ws/execute?token=eyJ0eXAi...");

// Option 2: First message with token
ws.send(JSON.stringify({
    id: "msg1",
    msg_type: "auth",
    channel: "execute",
    payload: {
        token: "eyJ0eXAiOiJKV1QiLCJhbGc..."
    }
}));
```

## Error Handling

AuthError variants:

| Error | Code | HTTP Status | Meaning |
|-------|------|-------------|---------|
| InvalidToken | 5001 | 401 | Token format invalid |
| TokenExpired | 5002 | 401 | Token has expired |
| AccessDenied | 5003 | 403 | Insufficient permissions |
| AuthenticationRequired | 5004 | 401 | Auth token required but missing |
| InvalidCredentials | 5005 | 401 | Wrong username/password |
| JwtError | 5006 | 401 | JWT library error |
| UserNotFound | 5007 | 404 | User doesn't exist |
| RoleNotFound | 5008 | 400 | Invalid role in token |

## Security Best Practices

1. **Secret Key Management**: Store JWT secret in environment variable or secure config file
2. **Token TTL**: Use short TTLs (1-2 hours) and implement refresh token flow for long-lived sessions
3. **HTTPS Only**: Always transmit tokens over HTTPS in production
4. **Token Rotation**: Rotate secret keys periodically
5. **Role Principle**: Use principle of least privilege - assign minimum required roles
6. **Audit Logging**: Log all auth events (success and failures)

```rust
let auth_manager = AuthManager::new(
    config.auth.secret_key.clone(),
    config.auth.token_ttl
);
auth_manager.register_default_policies();

// Share auth_manager across handlers (wrap in Arc<Mutex<>>)
let auth_manager = Arc::new(auth_manager);
```

### Scaling Considerations

- Token validation is stateless and horizontally scalable
