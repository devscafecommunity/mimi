## Test Files & Organization

### 1. `cli_integration_utils.rs` (Utility Infrastructure)

Foundational test framework providing:

#### Core Structures

- **`CliOutput`**: Captures CLI subprocess output (stdout, stderr, exit_code)
- **`CliProcess`**: Manages spawned CLI subprocess lifecycle
- **`TestContext`**: Manages test environment (auth, config, servers)
- **`TestServerConfig`**: Configuration for HTTP/WebSocket test servers

#### Key Helper Functions

```rust
run_cli_command(&["--version"])           // Execute CLI and capture output
spawn_cli_command(&["run"])                // Spawn CLI process (for streaming)
TestContext::new()                         // Create isolated test environment
ctx.create_admin_token()                   // Generate JWT for admin
ctx.create_user_token()                    // Generate JWT for user
ctx.create_guest_token()                   // Generate JWT for guest

```

```rust

```

## CLI Usage Reference

### Basic Commands

#### Version
```bash
mimi --version
mimi version
```

Output:
```
mimi 0.1.0
```

#### Help
```bash
mimi --help
```

Output shows all available commands and options.

#### Run with Config
```bash
mimi run --config /path/to/config.toml
```

If config file not found, uses default configuration.

### Config File Format

`config.toml`:
```toml
[server]
host = "127.0.0.1"
port = 8080
ws_port = 8081

[auth]
secret = "your-secret-key-here"
token_ttl = 3600

[logging]
level = "info"
```

All config fields are optional with sensible defaults.

### Authentication

#### Generate Token (Internal API)
```rust
let identity = Identity {
    user_id: "user-123",
    username: "alice",
    roles: {
        let mut set = HashSet::new();
        set.insert(Role::User);
        set
    },
};

let token = auth_manager.generate_token(&identity)?;
```

#### Validate Token (Internal API)
```rust
let identity = auth_manager.validate_token(&token_string)?;
```

#### Check Permissions (Internal API)
```rust
let has_permission = auth_manager.check_permission(
    &identity,
    "execute",
    "read"
)?;
```

## Adding New Test Cases

### Template: CLI Workflow Test

```rust
#[test]
fn test_my_new_command() -> Result<(), String> {
    let output = run_cli_command(&["my-command", "--flag"])?;
    assert_cli_success(&output)?;
    assert_output_contains(&output, "expected text")?;
    Ok(())
}
```

### Template: Auth/Backend Test

```rust
#[test]
fn test_my_auth_scenario() -> Result<(), String> {
    let ctx = TestContext::new()?;
    let token = ctx.create_admin_token()?;
    let identity = ctx.auth_manager.validate_token(&token)
        .map_err(|e| e.to_string())?;
    
    assert_eq!(identity.username, "admin");
    Ok(())
}
```

### Template: Error Handling Test

```rust
#[test]
fn test_my_error_scenario() -> Result<(), String> {
    let output = run_cli_command(&["invalid", "command"])?;
    assert!(!output.is_success());
    assert_output_contains(&output, "error message")?;
    Ok(())
}
```

## Known Limitations & Future Enhancements

### Current Limitations

1. **Server Mocking**: Tests don't spawn actual HTTP/WebSocket servers - they test CLI logic and auth layer in isolation
2. **Performance**: No load testing or concurrent client testing
3. **Network**: No simulation of actual network latency/timeouts
4. **Persistence**: All test state is in-memory (no database)

## Troubleshooting

### Test Hangs or Timeout

If tests hang:
```bash
# Run with timeout
timeout 30 cargo test -p mimi-cli --test 'cli_*'

# Run single-threaded for clarity
cargo test -p mimi-cli --test 'cli_*' -- --test-threads=1
```

### Build Failures with Ambiguous Re-exports Warning

This is expected - both `auth` and `ws` modules export `error` and `types`. Not critical.

```
warning: ambiguous glob re-exports
 --> crates/mimi-cli/src/lib.rs:8:9
```

### Auth Validation Errors

Ensure TestContext is created before generating tokens:

```rust
let ctx = TestContext::new()?;                    // ✅ Always do this first
let token = ctx.create_admin_token()?;            // ✅ Then generate token
```
